## 二、实际可用的功能（已实现且基本正确）

- ✅ 多线程分片下载（每任务最多 16 线程，按 Range 切片、写独立 `.part.N` 临时文件、完成后合并）
- ✅ 单线程下载（服务器不支持 Range 或小文件时降级）
- ✅ 暂停 / 取消 / 删除（cancel flag + 状态机）
- ✅ 进度/速度/状态实时回传（Tauri event `download-progress` / `download-completed` / `download-failed`）
- ✅ 下载队列、搜索、按状态筛选、批量删除（已完成）
- ✅ 设置面板 UI（目录选择走 `select_download_dir`，已正确接线）
- ✅ 下载完成自动打开文件夹 / 系统通知（前端监听 + 后端 `explorer.exe`）
- ✅ 拖拽 / 粘贴批量导入链接
- ✅ 磁盘空间预检（`get_available_space`，Windows 调用 `GetDiskFreeSpaceExW`）

## 三、关键缺陷（按严重程度排序）

### 🔴 P0 — 功能完全失效

**1. 浏览器扩展 → 桌面 App 集成是死的**
- 后端本地 HTTP 服务 `run_http`（`main.rs:63-134`）要求请求带 `X-Auth-Token` 头，且与随机生成的 token 完全相等才放行（`:91-100`）。
- 扩展 `background.js:35-44` 直接 `fetch('http://localhost:45678/download', …)` **不带任何 auth 头** → 永远返回 `401 Unauthorized`。
- 此外 `manifest.json` 声明了 `leledown` 协议处理器，但 `tauri.conf.json` 并未注册自定义 scheme，且 `background.js` 也从不用它。
- 结论：**从浏览器右键/扩展下发下载链接完全不可用**。

**2. 下载历史记录并未真正持久化**
- `db.rs` 完整实现了 SQLite 读写，但 `main.rs:39-43` 在启动时 `load_tasks()` 后**只打日志、不写回 `DownloadManager`**。
- 前端 `get_downloads`（`commands.rs:87-93`）读的是内存 map，重启后为空 → **重启即丢失全部历史**，历史库形同虚设。

**3. 断点续传是“伪功能”**
- `resume_download`（`download.rs:692-694`）只是再次调用 `start_download`，后者重新 HEAD、从字节 0 开始、覆盖写入。
- 没有任何“从已下载偏移量继续 Range 请求”的逻辑，也没有把 `downloaded_size` 持久化用于续传。
- 与 `SPEC.md` 宣称的“断点续传”不符。

### 🟠 P1 — 设置不生效 / 假选项

**4. 设置面板与后端完全脱钩**
- 后端 `update_download_settings` / `get_download_settings`（`commands.rs:140-163`）**从未被前端调用**（全仓 grep 确认，仅在 `SettingsPanel.tsx:63` 调用了 `select_download_dir`）。
- `SettingsPanel` 只把设置写入 `localStorage`，**不调用任何 Tauri 命令** → 改线程数/并发/速度限制对后端毫无影响。

**5. “同时下载任务数”“速度限制”是空开关**
- `DownloadConfig.max_concurrent`（`download.rs:128`）从未用于并发控制（无信号量/调度）。
- `speed_limit_kb`：`update_download_settings` 永远传 `0`（`commands.rs:154`），且 reqwest client 没有任何限流实现 → **限速功能不存在**。

**6. 线程数默认值前后不一致**
- 任务创建时硬编码 `thread_count: 16`（`download.rs:226`），后端 config 默认 16，设置 UI 默认 8 → 用户看到的和实际跑的对不上。

### 🟡 P2 — 正确性问题

**7. `open_folder` 校验误伤合法路径**
- `commands.rs:222-226` 拒绝包含 `(` 的路径。Windows `Program Files (x86)` 含 `(` → 若下载目录在此，打开文件夹会报错。
- 注：使用 `Command::new("explorer.exe").arg(&folder)`（非 shell）防注入是**正确的**，校验本身属于冗余且有害。

**8. 暂停/取消后残留 `.part.*` 临时文件**
- 暂停、失败后重试时，旧分片文件不会被清理（`download_chunk_to_temp` 仅在“取消中”删除自己的那片，`:662-664`），重新下载会生成新分片，旧片遗留占用磁盘。

**9. 文件名缺少安全过滤**
- `extract_filename`（`download.rs:729-749`）未清洗非法字符（`\ / : * ? " < > |`）与路径穿越，恶意 Content-Disposition / URL 可能写出越界文件或非法名。

## 四、安全与配置

- 🔴 **`"csp": null`**（`tauri.conf.json:25`）：Tauri 的 Content-Security-Policy 被关闭，建议至少设 `default-src 'self'`。
- 🟡 HTTP token 启动时以明文写入日志（`main.rs:66`），虽仅监听 `127.0.0.1` 不可远程访问，但属敏感信息落盘。
- 🟡 `schemars` 本地补丁（`vpatch/schemars-0.8.22/`，约 600 文件）是对上游 `indexmap` 双 cfg 缺陷的临时绕过，**仅影响 JSON Schema 属性顺序，不影响运行**——但需长期跟踪上游修复以移除补丁。

## 五、构建 / 打包配置不一致

- `installer-fix_2026-07-04.md` 记录：为绕过 Tauri NSIS 补丁损坏 PE 导入表（`TaskDialogIndirect` 报错），已改为 `targets: "app"` + 手写 makensis 打包。
- 但当前 `tauri.conf.json:30` **仍是 `"targets": ["nsis"]`**。按当前配置重新 `tauri build` 会复现那个损坏的安装包。建议要么把配置固定为可用的 `app` 方案并固化打包脚本，要么修复 NSIS 补丁步骤。

## 六、代码质量（中肯评价）

**做得好的地方**
- 下载引擎用独立 `.part.N` 文件避免共享写竞争，思路正确。
- 进度上报节流（300ms 才 emit）避免事件风暴。
- 取消用 `AtomicBool` + 提前 `drop(tx)`，逻辑干净。
- 前端组件拆分清晰，响应式布局（flex-wrap / min-w-0 / flex-shrink-0）处理得当。

**可改进的点**
- `db.rs` 中两处 `DownloadStatus → String` 的 match 可抽成统一函数，消除重复。
- `download_single` 在每个 chunk 都 `tasks.write()`（`:586`），频率偏高；可只在节流点更新。
- `format_bytes_simple`（Rust）与 `formatBytes`（TS）重复，可统一约定。
- `DownloadTask` 结构体偏大，部分字段（retry_count）多数时间为 0，略显冗余。

## 八、改进优先级建议

| 优先级 | 事项 | 工作量 |
|--------|------|--------|
| 高 | 修复扩展集成（共享 token 或注册 `leledown` 协议，并让扩展带鉴权） | 中 |

### 一句话总结
**核心下载引擎是扎实可用的，但“扩展集成、历史持久化、断点续传、设置生效、并发/限速”这五块要么完全失效、要么只是空壳——属于“看起来功能齐全、实际一半不能用”的状态，建议按上面的优先级逐项补齐。**
